# 🏢 วิเคราะห์และออกแบบระบบจัดการข้อมูลองค์กร (Tenant Settings)

การออกแบบระบบจัดการองค์กร (Tenant Settings) สำหรับระบบ SaaS แบบ Multi-tenant ถือเป็นหัวใจสำคัญ เพราะเป็นส่วนที่เจ้าของกิจการ (Admin) จะใช้ตั้งค่าเพื่อให้ระบบทำงานสอดคล้องกับธุรกิจของตนเอง

---

## 1. การวิเคราะห์สถาปัตยกรรม Web UI: แยก (Separated) vs รวม (Integrated)

การตัดสินใจว่าจะแยกหน้าจอ Settings ออกไปเป็นอีกเว็บ (เช่น `admin.domain.com`) หรือรวมไว้ในเว็บเดียว (เช่น `domain.com/settings`) มีข้อดีข้อเสียดังนี้ครับ:

### แบบที่ 1: รวมไว้ในระบบเดียวกัน (Integrated Web UI) - **⭐ แนะนำสำหรับ SME SaaS**
คือการเพิ่มเมนู **"ตั้งค่าองค์กร" (Settings)** ไว้ใน Sidebar หลัก โดยจะแสดงผลเฉพาะผู้ใช้งานที่มีสิทธิ์ระดับ Admin เท่านั้น
- **✅ ข้อดี:**
  - **Seamless Experience:** ผู้ใช้ไม่ต้องสลับหน้าเว็บไปมา สามารถตั้งค่าเลขที่เอกสาร แล้วกลับมาออกบิลได้ทันที
  - **Single Codebase:** ดูแลรักษาง่าย (Maintainability) พัฒนาและ Deploy เพียงแค่โปรเจกต์เดียว
  - **ลดความซับซ้อน:** ลูกค้า SME มักชอบระบบที่ครบจบในที่เดียว การมีหลาย Portal อาจทำให้เกิดความสับสน
- **❌ ข้อเสีย:**
  - ต้องเขียน Role-Based Access Control (RBAC) ในฝั่ง Frontend ให้รัดกุมมาก เพื่อซ่อนเมนูจากพนักงานทั่วไป
  - หากหน้าการตั้งค่ามีขนาดใหญ่มาก อาจทำให้ Bundle Size ของเว็บใหญ่ขึ้น (แก้ได้ด้วย Lazy Loading)

### แบบที่ 2: แยก Web UI ออกจากกัน (Separated Admin Portal)
คือการสร้าง Frontend Project ขึ้นมาใหม่แยกต่างหากสำหรับจัดการองค์กรโดยเฉพาะ
- **✅ ข้อดี:**
  - **Security:** ปลอดภัยสูงสุด เพราะแยกโค้ดและ Routing กันชัดเจน พนักงานทั่วไปไม่มีทางเข้าถึงหน้าเว็บนี้ได้
  - **Performance:** หน้าเว็บหลัก (ออกบิล) จะเบามาก เพราะไม่ต้องโหลดโค้ดส่วนการตั้งค่า
- **❌ ข้อเสีย:**
  - **Context Switch:** สร้างความรำคาญให้ User ที่เป็นเจ้าของกิจการ เพราะบางครั้งอยากตั้งค่าเล็กๆ น้อยๆ แต่ต้องสลับเว็บและล็อกอินใหม่
  - **Development Cost:** ต้นทุนในการพัฒนาและดูแลรักษาสูงขึ้น (ต้องตั้งค่า CI/CD, Nginx และ Domain เพิ่มเติม)

> **💡 บทสรุปข้อเสนอแนะ:** 
> สำหรับโปรเจกต์ **Senic Billing Next** ที่เน้นกลุ่ม SME แนะนำให้ใช้ **แบบที่ 1 (รวมไว้ในระบบเดียวกัน)** ครับ เพราะตอบโจทย์เรื่อง UX ที่ลื่นไหลกว่า ส่วนเรื่องความปลอดภัย เรามีระบบ JWT ที่แนบ Claim/Role และ `TenantId` ที่แข็งแกร่งอยู่แล้ว เพียงแค่ออกแบบ UI ให้แบ่งหมวดหมู่การตั้งค่าให้ชัดเจนก็เพียงพอครับ

## 2. ออกแบบโครงสร้างฟีเจอร์สำหรับโมดูล Tenant Settings

หากเรารวมเข้าไว้ในระบบเดียวกัน แนะนำให้ออกแบบหน้า Settings ให้เป็นแบบ **Tabs หรือ Side-navigation ย่อย** ภายในหน้าตั้งค่า โดยแบ่งโมดูลออกเป็นดังนี้:

### 🏢 2.1 ข้อมูลองค์กร (Company Profile)
ใช้สำหรับออกเอกสารต่างๆ (หัวกระดาษ)
- ชื่อบริษัท / องค์กร
- เลขประจำตัวผู้เสียภาษี (Tax ID)
- ที่อยู่, เบอร์โทรศัพท์, อีเมล, เว็บไซต์
- **อัปโหลดโลโก้บริษัท (Logo Upload)** เพื่อใช้แปะในไฟล์ PDF

### ⚙️ 2.2 การตั้งค่าเอกสาร (Document Defaults & Sequences)
- **เลขที่เอกสาร (Running Numbers):** ตั้งค่า Prefix (เช่น `INV-`), รูปแบบปี/เดือน (เช่น `YYYYMM`), และเลขรันปัจจุบัน
- **ตั้งค่าภาษีมาตรฐาน:** อัตรา VAT เริ่มต้น (เช่น 7%), รูปแบบ VAT เริ่มต้น (Inclusive/Exclusive), อัตราภาษีหัก ณ ที่จ่าย (WHT)
- **ข้อความหมายเหตุเริ่มต้น (Default Notes):** ข้อความท้ายบิล หรือเงื่อนไขการชำระเงินที่ต้องการให้ขึ้นอัตโนมัติในทุกบิล
- **ข้อมูลบัญชีธนาคาร (Bank Accounts):** สำหรับพิมพ์ลงในใบแจ้งหนี้เพื่อให้ลูกค้าโอนเงิน

### 👥 2.3 การจัดการผู้ใช้งานและสิทธิ์ (User & Role Management)
- **ระบบเชิญผู้ใช้งาน (User Invitations):** เชิญพนักงานเข้ามาในระบบผ่าน Email
- **จัดการบทบาท (Roles):** กำหนดว่าใครเป็น Admin (ทำได้ทุกอย่าง), Manager (ดูรายงานได้), หรือ Staff (ออกบิลได้อย่างเดียว)
- ระงับสิทธิ์การใช้งาน (Deactivate User) เมื่อพนักงานลาออก

### 🔌 2.4 การเชื่อมต่อระบบ (Integrations & API)
- **Payment Gateway (Omise):** ช่องสำหรับกรอก Public Key / Secret Key เพื่อเปิดใช้งาน Payment Link
- **การแจ้งเตือน (Notifications):** เชื่อมต่อ Line Notify เพื่อรับการแจ้งเตือนเมื่อมีคนชำระเงิน หรือเมื่อสต็อกสินค้าใกล้หมด

### 💎 2.5 ข้อมูลแพ็กเกจและการชำระเงิน (Subscription & Billing) - *[Optional สำหรับอนาคต]*
- แสดงแพ็กเกจปัจจุบัน (Free, Pro, Enterprise)
- สรุปปริมาณการใช้งาน (โควต้าเอกสารที่ออกไปแล้วในเดือนนี้)
- ประวัติการชำระเงินค่าบริการรายเดือน (SaaS Subscription)

## 3. Workflow การเข้าถึงหน้า Settings

1. **Menu Navigation:**
   - ที่มุมซ้ายล่างของ Sidebar จะมีเมนู ⚙️ **ตั้งค่า (Settings)**
   - หากผู้ใช้ไม่มี Role `Admin` เมนูนี้จะถูกซ่อน (ซ่อนผ่าน Zustand store / RBAC hook)
2. **Settings Layout:**
   - เมื่อคลิกเข้ามา จะเป็น Layout ที่มีเมนูย่อยด้านซ้าย (Profile, Documents, Users, Integrations) และพื้นที่แสดงฟอร์มตั้งค่าด้านขวา
3. **Save Action:**
   - เมื่อกดบันทึก จะยิง API `PUT /api/tenants/{tenantId}` หรือแยก Endpoint ตามโมดูลย่อย
   - ข้อมูลที่อัปเดต (เช่น โลโก้, รูปแบบภาษี) จะมีผลทันทีเมื่อไปหน้า "สร้างบิลใหม่"
